TS - A TypeScript module collection.

Preface

This solution is a collection of TypeScript modules. It is the successor of an earlier project called "TS". The earlier project is still available at github but not longer under development. The latest release of that earlier project was compiled using the TypeScript compiler in version 1.3.0.0. By the time of this writing, the current stable release of TypeScript is 1.8 and even Javascript is available in a newer version: "ECMAScript 2015". The new Javascript release came with some long requested features. One of them are the so called "Iteration protocols". It's a set of functions, keywords and conventions which make the iterator pattern (one of the GOF design patterns) an integral part of the language.

The iterator pattern is also the basic pattern which drives LINQ. The earlier project used it's own implementation of the iterator pattern. Since the iterator pattern is now an integral part of Javascript, that implementation became obsolet. There were also breaking changes in the TypeScript language which made it impossible to compile the old code with the new 1.8 TypeScript compiler. That gave me the motivation for a complete rewrite.

This projects starts with the basic infrastructure modules "Utils" and "Exception" as well as the standard LINQ operators in project "Linq" and the two most important collections (List and Dictionary) in project "Collections". The other modules which were available in the earlier project will follow in later releases.

Solution setup

The Solution is a multi project solution which contains one project for each namespace.

These projects are all based on the template. I decided to use this template, because there is no library template for TypeScript projects.

I removed all template files from the project directories and after that I got pretty much what I would call a library project.

The project directories are placed inside the solution directory. (Visual Studio tries to create the project directories at the same filehierarchy level as the solution directory. You have to select the target directory manually if you want to recreate the current solution setup.)

This setup has two adavantages.
First, it is much easyer to create the code documentation with TypeDoc when the projects are placed in a common root directory.
Second, you can start debugging right out of the IDE. You just have to select the "Test" project as your start project.

You could of course create a different setup without any problem. This setup is just the way I like to organize my source code. The "Docs" directory as well as the "Test" directory are not mandatory in order to compile the source code.

The solution is self contained. There is no reference to other projects outside this solution as well as there is no reference to third party libraries.

All projects in this solution have a common TypeScript setup. They are all set to compile to the ECMAScript version "ECMAScript 6". There is also no module system selected.

The "TS" project, which is the main project from the solution, has also the "Combine JavaScript output into file:" selected. The path in this setting is "H:\PROJECT\TYPESCRIPT\TS\TS.js" on my system. That is the path to the solution project "TS" and the filename is "TS.js". You may change the path or filename according to your own system.

Please keep in mind, that not all module system allow to compile into one file. So, if you select a module system appropriate for you own solution, you mal also change the latter described setting.

The other settings are not relevant for this solution. You may set them to whatever setting matches your own requirements best.

Projects

Collections

The "Collections" project comprises the two most important generic collection classes. Those are the "Dictionary" and the "List" class. You can read about the C# counterparts at MSDN.

Dictionary<TKey, TValue> | MSDN
List<T> | MSDN

The rest of the files in this project are the interfaces implemented by the latter described classes and a "KeyValuePair" implementation which is used by the Dictionary class. There is also an "Exception.ts" file which provides collection specific exception classes.

Exception

The "Exception" project comprises most of the exception classes defined in this solution. The project has only one file, the "Exception.ts" file. That file contains multiple class definitions. All exceptions in this file are bound to the "TS" namespace. The "TS.Exception" class defined in this project is the base class of all exception classes defined in this solution. The "Exception" class and all derived classes have a public "type" property which returns the fully qualified type of the exception as string. This way you can create type specific exception handlers and overcome the poor designed exception handling implemented in Javascript, which is based on the Javascript "Error" object.

Linq

The "Linq" project comprieses the classes and interfaces which build the foundation of the LINQ implementation. The "Extensions.ts" file contains all the extension functions which have a counterpart in the "Enumerable" class in .NET.

Enumerable | MSDN

In this implementation the functions are not bound to a class but are namespace functions. Some of the .NET LINQ extension functions are not implemented because of the limitations of the Javascript language.
Some others are no original LINQ functions as defined by Microsoft but none the less very useful. A lot of the functions have a function signature which differs from the original Microsoft definition. Those differences are also founded in the limitations of the Javascript language. All the functions provide a full set of Javascript documentation comments. There you will find a description of the  implementation specific differences. The VisualStudio IntelliSense function shows this comments while you type a function name. You should read those comments to prevent unexpected results. The following link leads to a full list of all implemented extension functions.

TS

The "TS" project is the solution project and just the root of the namespace in this solution. There is no class, interface or function definition at that level. You will find the "_reference.ts" file in that project which is referenced by most of the TypeScript files in this solution.

The directory of this solution project is also the output target for the combined Javascript file. The test setup requires the output file in that directory. If you choose to use a different output directory, you should also consider to change the references in the test files.

Utils

The "Utils" project is is a collection of functions which don't show a strong problem domain relation but are mainly solutions for general programming problems. Currently there are two files in that project. The "Utils.ts" and the "Assert.ts".

The "Utils.ts" file contains some conversion functions, parameter check functions, a GUID creator function and some more.

The "Assert.ts" file contains a number of assertion functions. Functions which take one argument and return a boolean, which tells you whether the argument value matches a specific condition or not.

The paraemter check functions in the "TS.Utils" namespace and the assertion functions in the "TS.Utils.Assert" namespace are heavily use throughout this solution.

Test

The "Test" project comprises the test files for the Collections, Exceptions, Linq and Utils projects. For obvious reasons there are no test files for the Docs, TS or Test project itself. The test file names follow a common conventions. Each file name starts with the project name which hosts the classes or functions which get tested in the current test. Separated by an underscore follows the class name the test belongs to. In case the test is designed to test functions which are not bound to a class (namespace functions) there won't be a class name. After that follows the ending 'test.ts" or "test_plain.js" which is also separated by an underscore. The files which end on "test.ts" are QUnit tests written in TypeScript. Those which end on "test_plain.js" are QUnit tests written in plain Javascript. The tests in plain Javascript became necessary because some tests couldn't be described in TypeScript. Each unit test has a corresponding HTML file wich is used to start the test.

Tere is also a HTML file called "TestSuite.html" which runs all the unit tests as a suite.
